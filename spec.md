# エンジニア教育管理システム 仕様書（検討中）

> このドキュメントは壁打ち相談中の仕様メモです。随時更新します。

---

## 1. システム概要

新卒エンジニアの教育・育成を管理するWebシステム。
チェックリストによる習得確認・S/A/B/C評価・課題提出と採点・進捗ダッシュボードを中心とする。
将来的に外部SaaS展開も視野に入れた設計とする。

---

## 2. ユーザーロール

| ロール | 説明 | 主な操作 |
|--------|------|---------|
| 受講者 | 対象エンジニア（新卒など） | チェックリスト確認、課題提出、進捗確認 |
| 講師 / メンター | 評価・課題レビュー担当 | 評価入力、課題採点、カリキュラム管理 |
| 管理者 | システム全体管理 | ユーザー管理、チェックリスト編集、全体把握 |
| HR担当者 | 採用・育成管理担当 | レポート閲覧、進捗ダッシュボード閲覧 |

---

## 3. グループ・組織構造

将来のマルチテナント（SaaS）を見据えた階層設計。

```
Tenant（テナント＝会社）        ← SaaS展開時に複数社対応（手動作成）
 └─ 年度コーホート              例：2024年度入社、2025年度入社
     └─ 部署・チーム            例：開発部、インフラチーム
         └─ 受講者              担当講師が割り当てられる
```

## 4. 認証・ユーザー招待

### 4.1 ログイン

- メールアドレス ＋ パスワード
- パスワードリセット機能あり（メール送信）

### 4.2 ユーザー招待フロー

```
管理者が招待メールを送信（メアド入力）
 ↓
受講者がメール内URLをクリック
 ↓
パスワード設定画面でアカウント作成
 ↓
ログイン可能
```

- 招待時にロールと所属（コーホート・部署）を管理者が設定
- 招待メールの有効期限あり（例：72時間）

### 5.2 評価基準

| 評価 | 基準 | スコア |
|------|------|-------|
| S | 指導なしで実行可能・他人に教えられる | 4点 |
| A | ほぼ自力で実行可能 | 3点 |
| B | 説明を受ければできる | 2点 |
| C | 理解不足・再教育必要 | 1点 |

| 時期 | 評価対象カテゴリ（目安） |

| 入社1ヶ月 | ①社会人基礎 |
| 入社3ヶ月 | ③Git / ④プログラミング基礎 |
| 入社6ヶ月 | ②IT基礎 / ⑤DB / ⑥Web開発 |
| 入社12ヶ月 | ⑦インフラ / ⑧開発プロセス（総合） |

## 6. 学習コンテンツ（ハイブリッド型）

各チェックリスト項目に「学習ガイド」を紐付ける。詳細コンテンツは外部リンクに委ねる。

```
チェックリスト項目
 └─ 学習ガイド（管理者・講師がMarkdownで編集可）
     ├─ 概要・学習のポイント（テキスト）
     ├─ 参考リンク（複数URL、タイトル付き）
     └─ 対応する課題・テスト（紐付け）
```

| 項目 | 内容 |
|------|------|
| 学習ガイド | 「何を・なぜ学ぶか」の概要をMarkdownで記述 |
| 参考リンク | Zenn記事・社内Notion・公式ドキュメント等のURL |
| 動画等 | 外部URL（YouTube・Udemy等）のリンクとして管理 |
| 閲覧ログ | **管理しない**（シンプルさを優先。将来拡張可） |

## 7. カリキュラムプラン

### 7.1 概要

受講者に「カリキュラムプラン」を割り当てることで、学習すべきチェックリスト項目と課題のセットを管理する。

```
CurriculumPlan（カリキュラムプラン）
 ├─ name（例：「新卒エンジニア基礎コース」）
 ├─ description
 └─ CurriculumItem（構成要素）
     ├─ ChecklistCategory（チェックリストカテゴリ）への紐付け
     ├─ Assignment（課題）への紐付け
     └─ 表示順
```

### 7.2 割り当て

- 管理者がユーザー or コーホートに対してプランを割り当てる
- 受講者はログイン後、自分のカリキュラムプランの一覧と各詳細を閲覧できる

## 8. 課題・テスト管理

### 8.1 課題タイプ（デフォルト）

| テスト | 内容 |
|--------|------|
| Git試験 | branch作成 + PR提出 |
| SQL試験 | 3テーブルJOIN |
| プログラム試験 | 簡易API作成 |
| デバッグ試験 | バグ修正 |

### 8.2 提出方法

- GitHubリポジトリURL
- テキスト記述回答
- （将来）ファイルアップロード

### 8.3 課題ルール

| 項目 | 設定 |
|------|------|
| 締め切り | あり（課題ごとに管理者・講師が設定） |
| 再提出 | 上限なし。提出回数のみカウント・記録 |

### 8.4 課題ステータスフロー

```
未着手 → 提出済 → レビュー中 → 評価完了
                               └─ 差し戻し → 提出済（再提出）
```

### 8.5 採点（AI ＋ 講師確認）

```
受講者が提出
 ↓
Claude API が自動評価（コード分析 or テキスト分析）
 ├─ AI評価コメント
 └─ 推奨スコア（合格 / 要再提出）
 ↓
講師に通知「〇〇さんが提出しました（AI評価済）」
 ↓
講師がAI評価を確認 → 承認 or 修正して確定
 ↓
受講者に通知「評価が完了しました」
```

- AIはあくまで**補助**。最終評価は必ず講師が確定する
- 講師画面にはAI評価と自分のコメント欄を並べて表示

## 9. 自己評価機能

### 9.1 方式：評価タイミング連動型

評価期間（1ヶ月 / 3ヶ月 / 6ヶ月 / 12ヶ月）が開始されたら、受講者が先に自己評価を入力し、その後に講師が評価する。

```
評価タイミング開始（管理者が手動で開始）
 ↓
① 受講者：各チェックリスト項目に自己評価（S/A/B/C）＋コメントを入力
 ↓
② 講師：受講者の自己評価を参照しながら講師評価（S/A/B/C）＋コメントを入力
 ↓
③ 双方がギャップ（差分）を確認 → 面談・フィードバックへ
```

### 9.2 ギャップ表示

| 項目 | 自己評価 | 講師評価 | 差分 |
|------|---------|---------|------|
| git branch | A | B | ↓ 過大評価 |
| SQL JOIN | C | B | ↑ 過小評価 |
| Docker基本 | B | B | ✓ 一致 |

- 過大評価・過小評価の件数をサマリで表示
- 面談の話題出しとして活用できる

### 9.3 制約

- 評価タイミング期間外は自己評価を入力できない（読み取りのみ）
- 講師が評価を完了するまで、受講者は講師評価を閲覧できない（先入観防止）

## 10. 通知機能

### 10.1 方式

システム内通知（ヘッダーのベルアイコン等）。メール・Slack等の外部連携はなし。
未読件数をバッジで表示し、通知一覧画面で確認・既読管理する。

### 10.2 通知トリガー一覧

| 受信者 | トリガー | 通知内容 |
|--------|---------|---------|
| 受講者 | 評価タイミング開始 | 「〇ヶ月評価が始まりました。自己評価を入力してください」 |
| 受講者 | 講師評価が完了した | 「講師が評価を入力しました。確認してください」 |
| 受講者 | 課題が差し戻された | 「課題『Git試験』が差し戻されました。フィードバックを確認してください」 |
| 講師 | 受講者が自己評価を完了した | 「〇〇さんが自己評価を入力しました。評価を入力してください」 |
| 講師 | 課題が提出された | 「〇〇さんが課題『SQL試験』を提出しました（AI評価済）」 |
| 管理者 | 未評価が〇日以上放置 | 「〇〇さんの評価が〇日間未入力です」 |

## 11. 進捗ダッシュボード

### 受講者向け
- カテゴリ別 S/A/B/C 分布（レーダーチャートなど）
- 課題提出状況・評価履歴
- 評価タイミングまでの残日数

### 講師向け
- 担当受講者一覧と進捗サマリ
- 未評価チェックリスト・未採点課題の通知
- 受講者比較

### 管理者 / HR向け
- 年度コーホート別・部署別の進捗集計
- 課題合格率・評価分布
- レポートエクスポート

## 13. 画面構成・権限設計

### 13.1 基本方針

- ロールごとにデフォルト権限を設定する
- **ユーザー単位で個別に権限を付与・剥奪できる**（管理画面のチェックボックスで管理）
- ロールのデフォルト権限をベースに、ユーザー個別の上書き設定を重ねる

```
ロールのデフォルト権限
 ＋ ユーザー個別権限オーバーライド（追加 or 剥奪）
 ＝ 最終的なそのユーザーの権限
```

## 16. データモデル（概要）

```
Tenant（テナント＝会社）※手動作成
 ├─ id
 ├─ name
 └─ plan

※ 以下の全テーブルは tenant_id を持つ（PostgreSQL RLS で保護）

User（ユーザー）
 ├─ tenant_id
 ├─ email / password_hash
 ├─ role: learner / instructor / admin / hr
 ├─ CohortYear（年度）
 └─ Department（部署）

UserPermissionOverride（ユーザー個別権限上書き）
 ├─ user_id
 ├─ permission_key（例: report.view, curriculum.edit）
 └─ granted: true / false

CurriculumPlan（カリキュラムプラン）
 ├─ name
 ├─ description
 └─ CurriculumItem（構成要素）
     ├─ checklist_category_id
     ├─ assignment_id
     └─ order

ChecklistTemplate（チェックリストテンプレート）
 └─ ChecklistItem（項目）× 100
     └─ LearningGuide（学習ガイド）
         ├─ body（Markdown本文）
         └─ ResourceLink（参考リンク）× 複数
             ├─ title
             └─ url

LearnerChecklist（受講者別チェックリスト）
 └─ LearnerChecklistItem
     ├─ self_rating: S / A / B / C（受講者入力）
     ├─ self_comment
     ├─ instructor_rating: S / A / B / C（講師入力）
     ├─ instructor_comment
     └─ evaluation_period_id

EvaluationPeriod（評価タイミング）
 ├─ type: 1m / 3m / 6m / 12m
 ├─ started_at（管理者が手動開始した日時）
 └─ LearnerEvaluation（評価サマリ）

Assignment（課題）
 ├─ title
 ├─ type: git / sql / program / debug
 ├─ deadline（締め切り）
 └─ Submission（提出物）
     ├─ github_url / text_answer
     ├─ submit_count（再提出回数カウント）
     └─ Review（採点結果）
         ├─ ai_comment
         ├─ ai_score
         ├─ instructor_comment
         ├─ final_score（数値: 合格 / 不合格 など）
         └─ status: ai_pending / instructor_reviewing / completed

Notification（通知）
 ├─ user_id
 ├─ type
 ├─ message
 ├─ link
 ├─ is_read
 └─ created_at
```
